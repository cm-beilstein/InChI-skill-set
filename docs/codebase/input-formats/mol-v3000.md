# MDL MOL V3000 Extended File Format

**Analysis Date:** 2026-04-22

---

## Overview

The **MDL MOL V3000** format (also known as Molfile V3000 or CTFile V3000) is an extended molecular file format developed by Molecular Design Limited (MDL, now part of Dassault Systèmes BIOVIA) that builds upon the V2000 format to support more complex chemical representations. Unlike V2000's fixed-column format, V3000 uses a keyword-based, free-format structure that provides much greater flexibility for representing sophisticated molecular features.

V3000 becomes necessary when the V2000 format reaches its limitations. Specifically, V3000 is required when: the molecule contains more than 999 atoms or 999 bonds (V2000's hardcoded limits); enhanced stereochemistry specifications are needed (absolute, relative, or racemic stereo collections); R-group logic for Markush structures is required; S-groups (structural groups) like repeating units, monomers, or copolymers need representation; or the molecule contains haptic bonds or star atoms used in organometallic complexes.

The InChI Software includes a full V3000 parser (`mol_fmt3.c`) that reads these extended molfiles and converts them to the internal representation used for InChI generation. The parser handles all V3000-specific features including enhanced stereo collections, S-groups, and the keyword-based atom/bond blocks.

## File Structure

```
+--------------------------------------------------+
|              HEADER BLOCK (3 lines)              |
|  Line 1: Molecule name                           |
|  Line 2: Comment/user information                |
|  Line 3: Program name + V3000 indicator          |
+--------------------------------------------------+
|           COUNTS LINE - V3000 (1 line)           |
|  V3000 AAABBB CCC DDD                            |
+--------------------------------------------------+
|         COLLECTION BLOCK (optional)                 |
|  MDLV30/STEABS n                                |
|  MDLV30/STEREL n                                |
|  MDLV30/STERAC n                                |
+--------------------------------------------------+
|              CTAB BLOCK                         |
|  +------------------------------------------+   |
|  |           ATOM BLOCK                      |   |
|  |  BEGIN ATOM                             |   |
|  |  index symbol x y z aamap CHG, RAD, ...  |   |
|  |  ...                                    |   |
|  |  END ATOM                               |   |
|  +------------------------------------------+   |
|  +------------------------------------------+   |
|  |            BOND BLOCK                     |   |
|  |  BEGIN BOND                              |   |
|  |  index type a1 a2 CFG TOPO...           |   |
|  |  ...                                    |   |
|  |  END BOND                               |   |
|  +------------------------------------------+   |
|  +------------------------------------------+   |
|  |         SGROUP BLOCK (optional)           |   |
|  |  BEGIN SGROUP                           |   |
|  |  ...                                    |   |
|  |  END SGROUP                             |   |
|  +------------------------------------------+   |
+--------------------------------------------------+
|       3D BLOCK (optional)                       |
|  BEGIN 3D ... END 3D                       |
+--------------------------------------------------+
|               M END                              |
+--------------------------------------------------+
```

## Header Block Detail

The V3000 header block follows the same 3-line structure as V2000, but Line 3 includes the version indicator.

### Line 1: Molecule name

**Position:** Columns 1-44 (max 44 characters)
**Contents:** The name of the molecule. Can include alphanumeric characters, spaces, and punctuation.
**Example:**
```
Caffeine
```

### Line 2: Comment/User Information

**Position:** Columns 1-44 (max 44 characters)
**Contents:** User-defined comments, software-specific metadata, or timestamps.
**Example:**
```
Generated by InChI Software v1.07
```

### Line 3: Program and Version Indicator

**Position:** Contains the program name (columns 1-34) followed by the V3000 version indicator
**Contents:** Must end with "V3000" to indicate this is a V3000 format file
**Example:**
```
InChI generation     1.0.0       V3000
```

The "V3000" marker at the end of Line 3 distinguishes V3000 files from V2000 files, which use "V2000" in the counts line instead.

## Counts Line V3000

The V3000 counts line differs significantly from V2000. Instead of fixed-column format with embedded counts, it uses keyword-based field parsing with the **V3000** marker.

### Format

```
V3000 aaabbb ccc ddd e
```

### Fields

| Field | Position | Description |
|-------|----------|-----------|
| **V3000** | Keyword | Version marker (identifies format version) |
| **aaabbb** | Integer | Number of atoms |
| **ccc** | Integer | Number of S-groups |
| **ddd** | Integer | Number of 3D constraints (for 3D coordinates) |
| **e** | Integer | Chiral flag (0 = not chiral, 1 = chiral) |

### Example

```
V3000 1247 1356 0 0 0
```

This indicates 1247 atoms, 1356 bonds, 0 S-groups, 0 3D constraints, and a non-chiral molecule.

### Pointers to Data Blocks

Unlike V2000 where the counts line also provides bond and property counts, V3000 separates the counts differently. The counts line provides pointers to where each data block begins:
- The number of atoms determines the ATOM block size
- The number of S-groups determines the SGROUP block
- The number of 3D constraints (if any) determines the 3D block
- Collections are specified separately in the COLLECTION block

## Collection Block Detail

The **Collection Block** is unique to V3000 and provides enhanced stereochemistry information. It is optional but required when absolute, relative, or racemic stereochemistry needs explicit specification.

### Format

```
Begin Collection
  MDLV30/STEABS n
  MDLV30/STEREL n
  MDLV30/STERAC n
End Collection
```

### Stereochemistry Types

#### MDLV30/STEABS - Absolute Stereo

Absolute stereochemistry defines each chiral center with an explicit R or S designation. All atoms in the collection have known absolute configurations.

**Format:**
```
MDLV30/STEABS n
  atom1 atom2 atom3 atom4
  atom1 atom2 atom3 atom4
  ...
```

Where `n` is the number of stereo groups following, and each line specifies 4 atoms defining a tetrahedral center.

**InChI Implementation** (from `mol_fmt.h`):
```c
#define MOL_FMT_V3000_STEABS 1
```

#### MDLV30/STEREL - Relative Stereo

Relative stereochemistry specifies groups that are relative to each other (e.g., "these two groups are on the same side" or "opposite sides").

**Format:**
```
MDLV30/STEREL n
  atom1 atom2
  ...
```

**InChI Implementation:**
```c
#define MOL_FMT_V3000_STEREL 2
```

#### MDLV30/STERAC - Racemic

Racemic mixtures specify groups where the configuration is unknown (equal mixture of enantiomers).

**Format:**
```
MDLV30/STERAC n
  atom1
  atom2
  ...
```

**InChI Implementation:**
```c
#define MOL_FMT_V3000_STERAC 3
```

### Example

```
Begin Collection
  MDLV30/STEABS 2
    1 2 3 4
    5 6 7 8
End Collection
```

This specifies two absolute stereo groups: atoms 1-2-3-4 and atoms 5-6-7-8 define tetrahedral centers.

## Atom Block V3000

The V3000 atom block uses a **keyword-based format** rather than fixed columns. Each atom consists of positional parameters followed by optional key-value property pairs.

### Structure

```
BEGIN ATOM
  index symbol x y z aamap
  CHG value
  RAD value
  CFG value
  MASS value
  VAL value
  HCOUNT value
  [other properties...]
  index symbol x y z aamap
  [properties...]
END ATOM
```

### Positional Parameters

| Field | Type | Description |
|-------|------|------------|
| **index** | Integer | Atom number (1-based) |
| **symbol** | String | Element symbol (1-3 characters, e.g., "C", "Fe", "Cl") |
| **x** | Float | X coordinate |
| **y** | Float | Y coordinate |
| **z** | Float | Z coordinate |
| **aamap** | Integer | Atom-atom mapping number (for reactions) |

### Key-Value Properties

After the positional parameters, optional properties can be specified as keyword-value pairs:

| Keyword | Type | Description |
|---------|------|------------|
| **CHG** | Integer | Formal charge (-5 to +5, or -1 for uncharged) |
| **RAD** | Integer | Radical type (0=none, 1=singlet, 2=doublet, 3=triplet) |
| **CFG** | Integer | Stereochemistry parity (1=odd, 2=even, 3=unspecified) |
| **MASS** | Integer | Isotopic mass (absolute value, e.g., 13 for C-13) |
| **VAL** | Integer | Valence (0=normal, -1=zero valence) |
| **HCOUNT** | Integer | Explicit hydrogen count |
| **STBOX** | Integer | Stereo box identifier |
| **INVRET** | Integer | Inversion/retention flag |
| **SUBST** | Integer | Substitution count |
| **UNSAT** | Integer | Unsaturation |
| **RBCNT** | Integer | Ring bond count |

### Atom Parsing in InChI

The InChI implementation reads atoms using the `MolfileV3000ReadAtomsBlock` function (`mol_fmt3.c`). The parser handles:

1. **Star atoms**: Atoms with symbol "*" are marked but ignored in standard processing
2. **Charge conversion**: V3000 charge values are converted to the V2000 convention (1=+3, 2=+2, 3=+1, 4=doublet, 5=-1, 6=-2, 7=-3)
3. **Isotopic mass**: Absolute isotopic mass is converted to the mass difference convention used internally
4. **Valence**: V3000 uses -1 for zero valence, converted to 15 for internal representation

### Example Atom Block

```
BEGIN ATOM
  1 C 0.0000 0.0000 0.0000 0
  2 C 1.4000 0.0000 0.0000 0
  3 C 2.1000 1.2124 0.0000 0 CHG 1
  4 O 3.3000 1.2124 0.0000 0 VAL -1
  5 N 1.4000 -1.2124 0.0000 0 MASS 14
END ATOM
```

- Atom 1: Carbon at origin
- Atom 2: Carbon at (1.4, 0, 0)
- Atom 3: Carbon with +1 charge
- Atom 4: Oxygen with zero valence
- Atom 5: Nitrogen with isotopic mass 14 (N-14)

## Bond Block V3000

The V3000 bond block uses keyword-based format similar to the atom block.

### Structure

```
BEGIN BOND
  index type a1 a2
  CFG value
  TOPO value
  index type a1 a2
  [properties...]
END BOND
```

### Positional Parameters

| Field | Type | Description |
|-------|------|------------|
| **index** | Integer | Bond number (1-based) |
| **type** | Integer | Bond type (1=single, 2=double, 3=triple, 4=aromatic) |
| **a1** | Integer | First atom number |
| **a2** | Integer | Second atom number |

### Bond Types

| Type | Description |
|------|-------------|
| 1 | Single bond |
| 2 | Double bond |
| 3 | Triple bond |
| 4 | Aromatic bond |
| 5 | Single or double (query) |
| 6 | Single or aromatic (query) |
| 7 | Double or aromatic (query) |
| 8 | Any bond (query) |

### Key-Value Properties

| Keyword | Type | Description |
|---------|------|------------|
| **CFG** | Integer | Bond stereo configuration (0=not stereo, 1=up, 4=either, 6=down) |
| **TOPO** | Integer | Topology (0=either, 1=ring, 2=chain) |

### Bond Parsing in InChI

The `MolfileV3000ReadBondsBlock` function (`mol_fmt3.c`) handles:
1. **Haptic bonds**: Bonds to star atoms (marked as haptic, skipped in standard processing)
2. **Stereo configuration**: CFG values are mapped to V2000 convention
3. **Topology**: TOPO values for ring/chain distinction

### Example Bond Block

```
BEGIN BOND
  1 1 1 2
  2 2 3 4
  3 1 2 3 CFG 1
  4 1 2 5 TOPO 1
END BOND
```

- Bond 1: Single bond between atoms 1-2
- Bond 2: Double bond between atoms 3-4
- Bond 3: Single bond 2-3 with stereo "up"
- Bond 4: Single bond 2-5 with ring topology

## Sgroup Block Detail

**Sgroups** (Structural Groups) in V3000 represent polymer repeating units, monomers, and other structural features that V2000 cannot encode.

### Sgroup Types

| Type Code | Description |
|----------|------------|
| **SUP** | Superatom (abbreviated group) |
| **MUL** | Multiple group (alternating/random copolymer) |
| **SRU** | Structure repeating unit |
| **MON** | Monomer |
| **COP** | Copolymer |
| **CRO** | Crosslink |
| **MER** | Mer (repeating unit) |
| **MOD** | Modification |
| **GRA** | Graft |
| **COM** | Component |
| **MIX** | Mixture |
| **FOR** | Formulation |
| **DAT** | Data Sgroup |
| **ANY** | Any polymer |
| **GEN** | Generic |

### Structure

```
BEGIN SGROUP
  id type subtype conn label
  SMT "subscript"
  ALIST atom1 atom2 atom3 ...
  BLIST bond1 bond2 bond3 ...
  SDI x1 y1 x2 y2
  [additional properties...]
END SGROUP
```

### Sgroup Fields

| Field | Description |
|-------|------------|
| **id** | Sgroup identifier (unique number) |
| **type** | Sgroup type (SUP, MUL, etc.) |
| **subtype** | Subtype for polymers (1=alternating, 2=random, 3=block) |
| **conn** | Connectivity (1=head-to-head, 2=head-to-tail, 3=either/unknown) |
| **label** | Sgroup label (unique identifier string) |
| **SMT** | Subscript (the text representation) |
| **ALIST** | Atom list members |
| **BLIST** | Bond list members |
| **SDI** | Bracket coordinates |

### InChI Handling

The InChI parser stores Sgroup data in the `MOL_FMT_SGROUP` structure (`mol_fmt.h`) but primarily processes the atom and bond information. Sgroups for polymers are recognized and the component atoms/bonds are included in the main structure.

### Example Sgroup

```
BEGIN SGROUP
  1 SRU 1 1 "POLY"
  SMT "n"
  ALIST 1 2 3 4
  BLIST 1 2
  SDI -1.0 0.0 1.0 0.0
  SDI -1.0 1.0 1.0 1.0
END SGROUP
```

This defines an SRU (structure repeating unit) with subscript "n" containing atoms 1-4 and bonds 1-2.

---

## Rgroup Block Detail

The **Rgroup Block** (optional) provides Markush logic for representing variable substituents in generic structures.

### Structure

```
BEGIN RGROUP
  rgroup_id
  ATOMS atom1 atom2 ...
  RBONDS bond1 bond2 ...
  [R logic specifications]
END RGROUP
```

### InChI Handling

The InChI software has limited support for R-groups. When encountered, the parser stores the R-group information but generates warnings that the structure may not fully represent all possible interpretations.

## Complete V3000 Example with Annotations

Below is a verified V3000 molfile example from the InChI test fixtures (enhanced_stereo_test_file_1.sdf):

```
Structure drawings enhanced stereochemistry.sdf
  ChemDraw01282608392D

  0  0  0     0  0              0 V3000
M  V30 BEGIN CTAB
M  V30 COUNTS 7 6 0 0 1
M  V30 BEGIN ATOM
M  V30 1 C -1.429260 -0.206189 0.000000 0
M  V30 2 C -0.714790 0.206311 0.000000 0
M  V30 3 O -0.714790 1.031557 0.000000 0
M  V30 4 C -0.000107 -0.206312 0.000000 0
M  V30 5 C 0.714577 0.206311 0.000000 0
M  V30 6 C 1.429260 -0.206312 0.000000 0
M  V30 7 N -0.000107 -1.031557 0.000000 0
M  V30 END ATOM
M  V30 BEGIN BOND
M  V30 1 1 1 2
M  V30 2 1 2 3 CFG=1
M  V30 3 1 2 4
M  V30 4 1 4 5
M  V30 5 1 5 6
M  V30 6 1 4 7 CFG=1
M  V30 END BOND
M  V30 END CTAB
M  END
```

### Detailed Annotation

| Line | Content | Explanation |
|------|---------|-------------|
| 1 | `Structure drawings enhanced stereochemistry.sdf` | Header line 1: Molecule name |
| 2 | `ChemDraw01282608392D` | Header line 2: Program + timestamp |
| 3 | (blank) | Header line 3: Comments (empty) |
| 4 | `0 0 0 0 0 0 V3000` | **Counts line**: Format version V3000, then atom/bond counts (will be in CTAB) |
| 5 | `M V30 BEGIN CTAB` | **CTAB Block**: Begin connection table |
| 6 | `M V30 COUNTS 7 6 0 0 1` | **Counts**: 7 atoms, 6 bonds, 0 atom lists, 0 Sgroups, 1 collection |
| 7 | `M V30 BEGIN ATOM` | **Atom Block**: Begin |
| 8-14 | Atom lines | Each atom: `M V30 id type x y z charge [properties]` |
| 8 | `M V30 1 C -1.429260 -0.206189 0.000000 0` | Atom 1: Carbon at (-1.43, -0.21, 0), charge 0 |
| 9 | `M V30 2 C -0.714790 0.206311 0.000000 0` | Atom 2: Carbon at (-0.71, 0.21, 0) |
| 10 | `M V30 3 O -0.714790 1.031557 0.000000 0` | Atom 3: Oxygen at (-0.71, 1.03, 0) |
| 15 | `M V30 END ATOM` | Atom block end |
| 16 | `M V30 BEGIN BOND` | Bond block start |
| 17-22 | Bond lines | Each bond: `M V30 id type a1 a2 [CFG=...]` |
| 17 | `M V30 1 1 1 2` | Bond 1: Single (1), atoms 1-2 |
| 18 | `M V30 2 1 2 3 CFG=1` | Bond 2: Single (1), atoms 2-3, CFG=1 (stereo config) |
| 23 | `M V30 END BOND` | Bond block end |
| 24 | `M V30 END CTAB` | CTAB end |
| 25 | `M  END` | Molecule end |

### Key V3000 Syntax Differences from V2000

1. **Counts line format**: V3000 has 7 fields vs V2000's 10 fields
2. **M V30 prefix**: Every V3000-specific line starts with `M  V30` (note two spaces)
3. **Keyword format**: Atoms and bonds use keyword format, not fixed columns
4. **Properties inline**: Charge, isotope, etc. appear after atom line as keywords (e.g., `CHG 1`)
5. **CFG for stereo**: Bond stereochemistry uses `CFG=1` or `CFG=2` syntax
6. **CTAB block**: Atom/bond counts are inside CTAB, not in main counts line
7. **Collection block**: Enhanced stereo groups use `BEGIN COLLECTION` / `END COLLECTION`

## Comparison: V2000 vs V3000

| Feature | V2000 | V3000 |
|---------|-------|-------|
| **Atom limit** | 999 | No practical limit |
| **Bond limit** | 999 | No practical limit |
| **Format** | Fixed-column | Keyword-based |
| **Sgroups** | Not supported | Full support |
| **Enhanced stereo** | Basic (atom parity only) | Collections (ABS, REL, RAC) |
| **R-groups** | Not supported | Supported |
| **Haptic bonds** | Not supported | Supported |
| **3D coordinates** | Limited | Full support |
| **Polymers** | Basic | Full polymer representation |

### When to Use Each Format

**Use V2000 when:**
- Molecule has fewer than 1000 atoms and 1000 bonds
- Standard stereochemistry (R/S centers, E/Z double bonds) is sufficient
- File size should be minimal
- Compatibility with older software is required

**Use V3000 when:**
- Molecule exceeds V2000 atom/bond limits
- Enhanced stereochemistry (absolute, relative, racemic) must be specified
- R-groups or Markush structures are needed
- Polymer repeating units need explicit representation
- Organometallic complexes with haptic bonds are being represented
- Complex stereochemistry requires grouping multiple centers

### V2000 Limitations

1. **Hardcoded limits**: Maximum 999 atoms and 999 bonds
2. **Fixed format**: Parsing errors if columns are misaligned
3. **Limited polymer support**: Cannot represent repeating units
4. **Basic stereo**: Only atom-level parity (no collections)
5. **No R-group logic**: Cannot represent variable substituents
6. **No haptic bonds**: Cannot represent coordinative bonds to metals

The InChI parser in `mol_fmt3.c` handles all these V3000 features, converting them to the internal representation used for canonicalization and InChI generation.
